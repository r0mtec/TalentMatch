# skill-recognition-service

Внутренний Laravel-сервис для rule-based распознавания навыков в тексте резюме.

## Назначение

Сервис получает plain text резюме, ищет в нем технологии и навыки по справочнику и синонимам, затем возвращает нормализованный список навыков кандидата.

У резюме нет обязательного шаблона. Сервис не должен ожидать фиксированные разделы вроде `skills`, `projects` или `experience`. Поиск выполняется по всему тексту.

ML/NLP на текущем этапе не используется. Подход простой: поиск слов и синонимов в тексте.

## Основные задачи

- принимать текст резюме после парсинга;
- нормализовать текст;
- искать технологии по словарю и синонимам;
- возвращать `normalized_name`, `raw_text`, `evidence_text`, `confidence`;
- фиксировать нераспознанные термины для будущего пополнения справочника;
- не принимать финальные решения по соответствию кандидата требованиям.

## Endpoint

- `POST /api/internal/skills/recognize`

Пример ожидаемого ответа:

```json
{
  "status": "done",
  "skills": [
    {
      "technology_id": null,
      "normalized_name": "laravel",
      "raw_text": "Laravel",
      "evidence_text": "PHP, Laravel, PostgreSQL",
      "confidence": 100
    }
  ],
  "unrecognized_terms": []
}
```

## Health endpoints

- `GET /health`
- `GET /ready`

## Зависимости

- справочник технологий и синонимов;
- текст, полученный от `document-parser-service`.

На MVP словарь может быть локальным массивом. Позже его лучше читать из PostgreSQL через `request-management-service` или отдельную read model.

## Текущее состояние

Сервис содержит простую реализацию `RuleBasedSkillRecognizer` с небольшим встроенным словарем:

- PHP;
- Laravel;
- PostgreSQL;
- React;
- Docker.

## Что нужно реализовать дальше

- загрузку словаря технологий и синонимов из БД;
- более аккуратную нормализацию текста;
- учет границ слов, чтобы не ловить ложные совпадения;
- сохранение evidence-фрагментов;
- возврат `technology_id`, когда навык найден в справочнике;
- снижение ложных совпадений при поиске по произвольному тексту резюме.
